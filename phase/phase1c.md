# phase1c.md — 이상형 조건 + 학교/회사 제외 설정

> Phase 1 의 마지막 조각. **매칭(Phase 2) 직전 준비.** 1B(코스+가용시간)가 끝나 있어야 한다.
> 1A/1B 보다 단순하다. 대부분 기존 테이블(`ideal_types`, `org_match_settings`)에
> 화면을 붙이는 작업. **네이버 API 불필요, 새 Edge Function 불필요.**
>
> - **PART 1**: 사람이 정할 것 (가벼움).
> - **PART 2**: Claude Code 구현. 각 Step `verify` 통과 후 진행.
>
> Claude Code 에게: PART 2 를 Step 1부터 순서대로. 1C 범위 밖(매칭 알고리즘/매칭 카드)은
> 만들지 않는다. 유료 잠금 조건은 UI·게이팅만 두고 실제 결제 연동은 하지 않는다.

---

## Phase 1C 목표

사용자가 **이상형 조건**(나이·키·흡연·음주·종교 등)을 설정하고,
**같은 학교/회사 제외** 여부를 토글할 수 있게 한다. 학벌/직장/연봉 같은
세분화 조건은 **유료(토큰) 잠금** 상태로 UI 만 노출한다(결제 연동은 Phase 2+).

## 완료 기준 (Definition of Done)

- [ ] 기본 이상형 조건(나이대·키·흡연·음주·종교)을 설정/수정/저장할 수 있다.
- [ ] 세분화 조건(학벌·직장·연봉)은 잠금 상태로 보이고, 잠금 안내가 뜬다(결제는 안 함).
- [ ] 같은 학교 제외 / 같은 회사 제외 토글이 동작하고 저장된다(기본 ON).
- [ ] 설정이 `ideal_types`, `org_match_settings` 에 반영되고 재진입 시 유지된다.
- [ ] "매칭 준비" 안내가 이상형/제외 설정까지 반영해 갱신된다.
- [ ] 모든 화면이 INVI 다크 테마(흰 본문)로 렌더된다.

# PART 1 — 사용자(본인)가 정할 것

### 1. 환경 켜두기
Docker → `supabase start` → `npx expo start`. (1C 는 Edge Function 불필요라
`functions serve` 는 안 켜도 됨)

### 2. 기본 / 잠금 조건 경계 확인
아래 경계로 갈지만 확인. (이전 기획대로)
- **무료(기본) 조건**: 나이대, 키 범위, 흡연, 음주, 종교
- **유료 잠금 조건**: 학벌, 직장(산업/규모), 연봉대 → 1C 에선 **잠금 UI 만**, 결제 X
- **같은 학교/회사 제외**: 기본 **ON**. 둘 중 한 명이라도 ON 이면 매칭 제외(매칭 로직은
  Phase 2 에서 적용. 1C 는 설정값 저장까지).

### 3. 이상형 미설정 시 정책 확인
- 이상형을 아무것도 설정 안 해도 가입/매칭 준비는 가능(조건 없음 = 제약 없음).
  필수 아님. (확인만)

# PART 2 — Claude Code 구현 작업

## Step 1. 이상형 입력 옵션 정리

`src/features/profile/` 또는 기존 옵션 상수 재사용:
- 나이대: min/max (예: 23~40), 키 범위: min/max(cm).
- 흡연·음주·종교: 1A 와 동일 옵션 + **"상관없음"** 옵션 추가(이상형은 미지정 허용).
- 세분화(잠금): 학벌·직장·연봉 옵션은 **표시는 하되 비활성/잠금** 처리.

**verify:** 옵션 상수·타입이 정의되고 에러 없다.

## Step 2. 기본 이상형 설정 화면

`app/(tabs)/ideal-type.tsx` (또는 설정 하위 라우트):
- 나이대(슬라이더/범위), 키 범위, 흡연/음주/종교(각 택1, "상관없음" 포함).
- 미지정 = 조건 없음(매칭 시 제약 안 함)으로 처리.
- INVI 톤, 다크 테마, theme 토큰.

**verify:** 각 항목 설정 후 저장 → 재진입 시 유지.

## Step 3. 이상형 저장 (RPC 우선)

`update_ideal_type(...)` RPC (security definer, 본인만) — 1A 의 `update_profile`
패턴과 동일하게:
- `ideal_types` 에 부분 업데이트(전달된 값만).
- 범위 검증(min ≤ max, 키/나이 합리적 범위).
- 잘못된 enum 거부.
> 단순 저장이라 RPC 가 과하면 RLS 하의 클라이언트 update 도 허용. 단 1A 에서
> 쓰기를 RPC 로 통일했으면 일관성 위해 RPC 로. (의미 있는 저장은 RPC, 사소한
> 토글은 클라이언트 — 1A 와 동일 기준)

**verify:** 저장이 `ideal_types` 에 반영되고 범위 오류는 거부된다.

## Step 4. 세분화 조건 — 잠금 UI (결제 X)

- 학벌/직장/연봉 조건을 **잠금 상태**로 표시.
- 탭하면 안내: "토큰으로 잠금 해제하면 학벌·직장·연봉 조건으로 매칭할 수 있어요"
  (실제 결제·잠금해제는 Phase 2+ 에서. 여기선 안내 모달까지만).
- `ideal_types` 의 `school_pref`/`company_pref`/`salary_pref`/`paid_until` 은
  건드리지 않음(잠금 상태 유지).

**verify:** 잠금 조건은 설정 불가 상태로 보이고, 안내만 뜬다. DB 미변경.

## Step 5. 같은 학교/회사 제외 토글

`app/(tabs)/` 설정 영역 + `org_match_settings`:
- "같은 학교 제외" 토글, "같은 회사 제외" 토글. **기본 ON**.
- 변경 시 `org_match_settings` 에 저장(없으면 생성).
- 안내 문구: "둘 중 한 분이라도 켜면 같은 학교/회사끼리는 매칭되지 않아요.
  동문/동료를 만나고 싶으면 끄세요."
> 단순 토글이라 클라이언트 update 허용(RLS 본인만). 또는 작은 RPC.

**verify:** 토글 on/off 가 저장되고 재진입 시 유지된다. 기본값 ON 확인.

## Step 6. 설정 허브 / 진입 정리

- 프로필·이상형·가용시간·제외설정으로 가는 **설정 화면**(또는 탭) 정리.
- 흩어진 진입점을 한 곳에서 접근 가능하게(과한 재구성 금지, 진입 링크만 모음).

**verify:** 한 화면에서 프로필/이상형/가용시간/제외설정에 모두 접근된다.

## Step 7. 매칭 준비 안내 최종 갱신 (표시만)

- "매칭 준비 완료" 조건을 1A·1B·1C 종합으로 정리:
  - 필수 프로필 + 자기소개 3개 + 태그 5개 (1A)
  - 학교/직장 인증 1개 (1A)
  - 공개 코스 1개 + 가용시간 1개 (1B)
  - 이상형/제외 설정은 **선택**(미설정도 준비 완료 가능).
- 미충족 항목을 구체적으로 안내.
> 실제 매칭 시작은 Phase 2. 1C 는 안내 문구 확정까지.

**verify:** 조건 충족/미충족에 따라 안내가 정확히 바뀐다.

## 작업 순서 요약

```
[사람]   환경 켜기 + 기본/잠금 조건 경계 확인

[Claude] 1 옵션 정리 → 2 이상형 화면 → 3 이상형 저장(RPC)
        → 4 세분화 잠금 UI → 5 학교/회사 제외 토글
        → 6 설정 허브 → 7 매칭 준비 안내 최종
```

## 보안 메모 (CLAUDE.md 와 일치)

- 의미 있는 저장은 security definer RPC, 사소한 토글은 RLS 하 클라이언트 update.
- 같은 학교/회사 제외 판정은 Phase 2 매칭에서 `org_name_hash` 비교로(평문 미사용).
- 잠금(유료) 조건은 1C 에서 DB 변경·결제 연동 없이 UI·안내만.

## Phase 1C 종료 후 → Phase 1 전체 완료

Phase 1(프로필·인증·코스·가용시간·이상형·제외설정)이 끝난다. 다음은 **Phase 2 (매칭)**:
- 매칭 Edge Function: Hard Filter(같은 학교/회사 제외 등) → Scoring(코스/시간/거리/
  이상형/MBTI/매너) → Selection(가중 랜덤 1명)
- 매일 13시/23시 매칭 배치, 매칭 카드 UI(얼굴·학력·직장 블러), 양방향 관심
- 학력/직장 블러 양방향 공개, 토큰 결제 연동(잠금 조건 해제 포함)

> 막히거나 명세가 모호하면 멈춰서 질문할 것. (CLAUDE.md 가이드라인 1번)
