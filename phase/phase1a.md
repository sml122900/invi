# phase1a.md — 프로필 본문 + 학교/직장 인증

> Phase 1 의 첫 조각. **네이버 API 불필요** (1B에서 사용).
> Phase 0 가 끝나 있어야 한다(가입/로그인/온보딩 동작 + 17개 테이블).
>
> - **PART 1**: 사람이 미리 정할 것 (가벼움).
> - **PART 2**: Claude Code 가 구현. 각 Step 의 `verify` 통과 후 다음으로.
>
> Claude Code 에게: PART 2 를 Step 1부터 순서대로. 각 `verify` 통과 후 진행.
> Phase 0 에서 `vault` 스키마가 Supabase 내부 충돌로 `private` 로 변경됐다.
> 인증 테이블은 그 스키마의 `verifications` 를 사용한다(실제 스키마명은 코드에서 확인).
> 1A 범위 밖(코스/가용시간/이상형/매칭)은 만들지 않는다.

---

## Phase 1A 목표

가입·온보딩을 마친 사용자가 **매력적인 프로필 본문을 작성**하고,
**학교 또는 직장을 이메일로 인증**할 수 있게 한다. (얼굴/학력/직장은 여전히 비공개)

## 완료 기준 (Definition of Done)

- [ ] 프로필 본문(얼굴상·MBTI·흡연·음주·종교·키·지역·자기소개·태그)을 입력/수정할 수 있다.
- [ ] 자기소개는 prompt 선택형으로 3~6개 답변할 수 있다.
- [ ] 프로필 완성도(%)가 계산되어 표시된다.
- [ ] 학교 또는 직장 이메일로 인증 코드를 요청 → 입력 → 검증까지 동작한다.
- [ ] 인증 성공 시 `verifications` 에 도메인 해시가 기록된다(평문 도메인은 매칭에 안 쓰임).
- [ ] 인증 정보는 프로필 화면에 "인증됨"으로만 표시되고, 학교/직장명은 노출되지 않는다.
- [ ] 모든 화면이 INVI 다크 테마(흰 본문)로 렌더된다.

# PART 1 — 사용자(본인)가 정할 것

### 1. 환경 켜두기
Phase 0 와 동일: Docker Desktop 실행 → 프로젝트 루트에서 `supabase start` →
`npx expo start`. (USB 테더링 + LAN IP `10.75.58.41` 유지)

### 2. 테스트용 이메일 준비
1A 의 이메일 인증은 **로컬에선 실제 메일을 발송하지 않고** 인증 코드를 화면/DB로
확인하는 방식이다(아래 Step 7 설명). 그러니 실제 학교/회사 메일이 없어도 된다.
테스트 시 도메인만 그럴듯하게 입력하면 된다. 예:
- 학교 테스트: `test@yonsei.ac.kr`
- 직장 테스트: `test@samsung.com`

### 3. 인증 도메인 정책 (MVP 기준 — 확인만)
- **학교**: 이메일 도메인에 `ac.kr` 포함 → 학교로 인정 (예: `*.ac.kr`)
- **직장**: 그 외 도메인 → 직장으로 인정 (단 `gmail/naver/daum/kakao` 등 범용 메일은 거부)
- 특정 학교/회사 화이트리스트는 나중(베타)에. MVP 는 위 규칙으로 단순화.
> 이 정책으로 갈지만 확인해주면 된다. 바꾸고 싶으면 알려달라.

# PART 2 — Claude Code 구현 작업

## Step 1. 프로필 데이터/입력값 정리

`profiles` 테이블의 1A 대상 컬럼을 확인하고, 입력 옵션 상수를 `src/features/profile/`
또는 `src/utils/` 에 정의한다.

- `height_cm`(int), `region`(거주 구), `active_regions`(text[]),
  `mbti`(EI/NS/TF/JP), `face_type`(동물상), `smoking`/`drinking`(4단계),
  `religion`, `intro_prompts`(jsonb: [{prompt, answer}]), `tags`(text[]),
  `completeness`(int).
- 선택 옵션 상수:
  - 얼굴상: 강아지/고양이/여우/토끼/사슴/곰/늑대/다람쥐 (성별 무관 공통 세트로 시작)
  - 흡연·음주: none/sometimes/social/often (한국어 라벨 매핑)
  - 종교: 선택안함/무교/기독교/천주교/불교/기타
  - 자기소개 prompt 후보(택3~6): "최근 가장 행복했던 순간은…", "주말엔 보통…",
    "데이트 상대에게 양보 못 하는 한 가지는…", "친구들이 말하는 나는…",
    "요즘 빠진 콘텐츠는…", "함께 가보고 싶은 곳은…"
  - 취향 태그: 음악/영화/운동/여행/음식/콘텐츠 카테고리별 칩

**verify:** 옵션 상수가 정의되고 타입 에러가 없다.

## Step 2. 프로필 기본 정보 입력 화면

`app/(tabs)/profile.tsx` (또는 별도 편집 라우트) + `features/profile/`:
- 키(슬라이더/숫자), 거주 구, 활동 지역(복수 선택), 얼굴상(택1),
  MBTI(택1), 흡연/음주(각 택1), 종교(택1).
- INVI 톤. 다크 테마. theme 토큰만 사용.

**verify:** 각 항목을 고르고 저장하면 `profiles` 에 반영된다(재진입 시 유지).

## Step 3. 자기소개 prompt

- prompt 후보 중 **3개 이상** 골라 각각 답변(150~300자) 입력.
- 저장은 `intro_prompts` 에 `[{prompt, answer}]` jsonb 로.
- 빈 답변/길이 초과 검증.

**verify:** 3개 작성 후 저장 → 재진입 시 그대로 보인다. 2개 이하면 안내.

## Step 4. 취향 태그

- 카테고리별 칩에서 5~10개 선택 → `tags` 배열로 저장.

**verify:** 선택/해제가 저장되고 유지된다.

## Step 5. 프로필 저장 + 완성도 계산 (RPC)

프로필 저장과 완성도 계산을 **RPC 로** 처리한다(클라이언트 직접 update 대신 일관).
- `update_profile(...)` RPC (security definer, 본인만):
  - 전달된 필드를 `profiles` 에 update.
  - `completeness` 재계산: 핵심 필드(키·지역·얼굴상·MBTI·흡연·음주) +
    자기소개 3개 + 태그 5개 충족 비율을 0~100 으로.
  - 잘못된 enum/길이 등은 서버에서 거부.
> 단순 컬럼 update 가 많아 RPC 가 과하면, profiles 의 RLS(본인 update 허용) 하에
> 클라이언트 update + 완성도만 트리거/함수로 계산해도 된다. 일관성·검증 이점을
> 고려해 RPC 를 우선 검토하되, 과하면 후자를 택하고 이유를 남긴다.

**verify:** 저장 시 완성도(%)가 갱신되어 화면에 보인다.

## Step 6. 프로필 미리보기 (본인 시점)

- 입력한 내용이 "매칭 카드에서 이렇게 보인다" 형태로 미리보기.
- 얼굴상·나이·키·MBTI·지역·태그·자기소개. **학교/직장은 '인증됨'(블러 표현)으로만.**

**verify:** 미리보기에 본문이 정상 표시되고, 학교/직장 평문은 어디에도 안 보인다.

## Step 7. 학교/직장 인증 — 코드 요청 (RPC, 로컬은 코드 노출)

**설계 핵심**: 이메일 발송 인프라(Resend 등)는 베타에서 붙인다. 로컬·MVP 에선
인증 코드를 **이메일로 보내는 대신** 안전한 범위에서 확인 가능하게 한다.

`request_org_verification(p_email text, p_type text)` RPC (security definer):
1. `p_type` 은 'school' | 'company'. `p_email` 형식 검증.
2. **도메인 추출** 후 정책 적용:
   - `ac.kr` 포함 → school 로만 허용.
   - 범용 메일(gmail/naver/daum/kakao/hotmail/outlook 등) → 거부('generic_email').
   - 그 외 → company 허용.
   - `p_type` 과 판정이 불일치하면 거부.
3. 6자리 코드 생성 → **코드 해시**를 `verifications` 에 status='pending' 으로 저장
   (`user_id`, `type`, `org_name_hash = sha256(도메인)`, `email_hash = sha256(email)`,
   코드 해시, 만료 `now()+10분`). 기존 pending 있으면 대체.
4. 분당 요청 횟수 제한(예: 1분 1회) — 남용 방지.
5. **개발 모드 전용 코드 노출**:
   - `config` 플래그(예: `app.dev_mode` GUC 또는 별도 설정 테이블)가 true 일 때만
     생성된 평문 코드를 반환한다.
   - **프로덕션에선 절대 코드를 반환하지 않는다**(이메일로만). 이 분기를 코드에 명확히.
   > 로컬에선 RPC 응답의 코드(또는 Supabase Studio 에서 pending 행)로 다음 단계 진행.

**verify:** 학교/직장 이메일로 요청 시 pending 행이 생기고(개발 모드면 코드 확인 가능),
범용 메일·형식 오류·타입 불일치는 적절한 에러로 거부된다.

## Step 8. 학교/직장 인증 — 코드 확인 (RPC)

`confirm_org_verification(p_code text)` RPC (security definer):
1. 본인의 최신 pending 인증을 찾는다. 만료(10분) 지났으면 'expired'.
2. 입력 코드 해시와 저장 해시 비교. 불일치 횟수 누적(예: 5회 초과 시 차단).
3. 성공 시 status='verified', `verified_at=now()`, `reverify_due=now()+6개월`.
4. **org_name 평문은 저장하지 않거나 표시에만 쓰고, 매칭 비교는 `org_name_hash` 만.**

**verify:** 올바른 코드 → 'verified' 로 전환. 틀린/만료 코드 → 적절한 에러.
완료 후 프로필에 "학교 인증됨"/"직장 인증됨" 배지가 뜬다(조직명 비노출).

## Step 9. 인증 화면 UI

`app/(onboarding)/` 또는 `app/(tabs)/` 에 인증 플로우 화면:
- 학교/직장 택1 → 이메일 입력 → "인증 코드 받기"(request RPC)
- 코드 입력 → "확인"(confirm RPC) → 성공 배지
- 로컬 개발 편의: 개발 모드일 때 받은 코드를 화면에 잠깐 노출하거나,
  "Studio 에서 확인" 안내. (프로덕션 빌드에선 이 노출 제거)

**verify:** 화면에서 요청→입력→인증 완료까지 한 흐름으로 동작한다.

## Step 10. 매칭 활성화 조건 점검 (표시만)

- 1A 기준 "매칭 준비" 상태를 계산해 프로필/홈에 안내(아직 매칭 시작은 안 함):
  - 필수 프로필 + 자기소개 3개 + 태그 5개 + 학교/직장 인증 1개 완료 시
    "매칭 준비 완료" 표시.
  - 미충족 항목을 안내(예: "자기소개 1개 더 작성하면 준비 완료").
> 실제 매칭은 Phase 2. 여기선 안내 문구만.

**verify:** 조건 충족/미충족에 따라 안내가 정확히 바뀐다.

## 작업 순서 요약

```
[사람]   환경 켜기 + 도메인 정책 확인 + 테스트 이메일 준비

[Claude] 1 옵션 상수 → 2 기본정보 → 3 자기소개 → 4 태그
        → 5 저장+완성도(RPC) → 6 미리보기
        → 7 인증 코드 요청(RPC) → 8 코드 확인(RPC) → 9 인증 UI
        → 10 매칭 준비 안내
```

## 보안 메모 (CLAUDE.md 와 일치)

- 인증 코드는 **해시로만 저장**, 만료·시도횟수 제한. 평문 코드는 개발 모드에서만 노출.
- 학교/직장 식별은 **도메인 해시(`org_name_hash`)** 로만. 평문 도메인/조직명은
  매칭 로직에 끌어오지 않는다.
- 잔액·검증 등 민감 변경은 security definer RPC 로만(클라이언트 직접 update 지양).
- 로그/에러에 이메일·도메인 등 식별정보를 남기지 않는다.
