### Phase 1A 후속 — `app.dev_mode` GUC → `app_config` 테이블

**문제**
`alter database postgres set "app.dev_mode" = 'true'` 가 로컬 Supabase 의 `postgres` 역할 권한으로 실패. `supabase db reset` 시 seed.sql 실행이 끊김.

**해결**
GUC 대신 `public.app_config (key text primary key, value text)` 테이블 도입. RLS 활성 + security definer 함수만 접근. `request_org_verification` RPC 가 `select value from app_config where key='dev_mode'` 결과를 `coalesce` 로 fallback 처리.
- 로컬: seed.sql 이 `dev_mode='true'` 삽입 → 코드 반환
- 프로덕션: 행 없음 → coalesce false → 코드 미반환

`20260524000001_app_config.sql` 마이그레이션으로 정리.

- 기존 `delete-then-insert` 가 두 분리 호출 → 중간 실패 시 사용자가 가용 시간 통째로 잃음
- `set_available_times(p_slots jsonb)` security definer RPC 신설
  - `auth.uid()` null 가드, jsonb 배열 형식 검증
  - 각 슬롯 형식 검증: `jsonb_typeof` 로 타입, `day_of_week` 0–6, `slot` enum 3종
  - `delete + insert` 가 함수 본문의 단일 트랜잭션 안에서 실행 — 실패 시 전체 롤백
  - 본인 한정: `delete where user_id = v_uid`, `insert select v_uid, ...`
- 클라이언트 시그니처 그대로 유지, 내부만 RPC 호출 + 에러 throw 로 교체
- `supabase db reset` 으로 마이그레이션 6개(이 시점) 모두 깨끗하게 적용 확인

**M2: `database.ts` 정식 재생성 + 1C 시그니처 nullable 보정 마이그레이션**
- `supabase gen types typescript --local` 로 재생성 시 `update_ideal_type.Args` 가 non-null required 로 잡힘
  - 1C 원본 마이그레이션이 `default null` 누락 → gen types 가 "default 없으면 required" 로 추론
- 1A 의 `update_profile` 패턴(`p_x int default null`)과 일관성 위해 새 마이그레이션 `20260524000005_update_ideal_type_nullable_args.sql` 추가 → `create or replace` 로 시그니처에 `default null` 명시, 본문은 1C 원본과 동일
- 우리 원칙: **이미 적용된 마이그레이션은 immutable, 보정은 새 파일로** (결정 기록: `docs/decisions/migration-immutability.md`)
- 재생성 후에도 TS `null !== undefined` 충돌 잔존 → 1A 와 동일하게 `Record<string, unknown>` 우회 패턴 적용
- 트러블슈팅 기록: `docs/troubleshooting/supabase-gen-types-null-vs-undefined.md`
